01

Minha rotina — como e um dia real

O ritmo

Abro o Claude Code umas 13 vezes por dia. Cada sessao dura entre 10 minutos e 2 horas. Em media, dou 8-9 instrucoes por sessao. Em 15 dias, foram 464 entregas feitas.

Nao e que eu trabalho rapido. E que eu montei um sistema onde eu dou a direcao e o Claude executa. Meu trabalho e decidir O QUE fazer, nao COMO fazer.

Como comeca uma sessao

Abro o terminal e digo algo assim:

Continuar com o Grupo 2 do backlog. Issue #201. Repo em ~/Claude/GitHub/raiz-platform

Ou:

Tem um bug no modulo de contratos — campo businessUnitId nao reconhece camelCase. Issue #312. Diagnosticar antes de mexer.

Eu nao explico como fazer. Digo o que quero e onde esta. O sistema decide qual assistente usar, qual abordagem seguir, e executa.

O que eu faco enquanto ele trabalha

Se e algo simples, fico olhando e corrijo se desviar. Se e algo grande, mando rodar em background e vou para outra sessao. Ja cheguei a ter 3-4 sessoes rodando ao mesmo tempo em projetos diferentes.

Como termina uma sessao

Estou parando por hoje. Salva onde paramos, o que foi feito, o que falta.

Na proxima vez:

Continuando de ontem. Le o estado salvo e me diz onde paramos.

E ele retoma exatamente de onde parou.

02

Como eu peco as coisas

💡

A qualidade do que o Claude entrega depende 100% de como voce pede. Ele faz exatamente o que voce diz — e esse e o problema. Se voce pede errado, ele executa errado, rapido, e com confianca.

Exemplos reais do que funciona

Para resolver uma issue do backlog:

Resolver issue #201. Ler a issue, criar uma especificacao, implementar, verificar contra a especificacao, rodar testes, criar PR linkada. Repo em ~/Claude/GitHub/raiz-platform

Para corrigir um bug:

Bug: quando clica em salvar no formulario de perfil, o telefone nao salva. Endpoint /api/profile. ANTES de mexer em codigo, investiga a causa. Me mostra o que encontrou antes de implementar.

Para construir algo novo:

Quero um endpoint que recebe uma lista de alunos e retorna quais estao presentes. Usar a tabela attendance do Supabase. Testar com dados reais.

O que NAO funciona

❌

“Corrige o bug” (qual bug? onde?)

“Melhora o codigo” (qual parte? melhora como?)

“Implementa a feature” (qual? como deve funcionar?)

Quanto mais vago, pior o resultado. Diga O QUE, ONDE, e COMO VERIFICAR.

Quando interromper

Se voce ve que ele esta indo na direcao errada, interrompa imediatamente:

Para. Nao e isso. O problema nao e no endpoint, e no banco. Investiga a query.

Cada minuto que voce espera ele seguir no caminho errado e um minuto de retrabalho.

Pedidos com pontos de salvamento

Para qualquer coisa que leve mais de 15 minutos:

Implementar features A, B e C. Salvar o progresso depois de cada uma. Rodar testes antes de salvar. Se algo falhar, para e me avisa antes de tentar consertar.

03

Como eu gerencio o que precisa ser feito

GitHub Issues como meu backlog

Tudo que precisa ser feito vira uma issue no GitHub. Cada issue tem um titulo claro, uma descricao do que precisa acontecer, e etiquetas de prioridade. Eu organizo em grupos por prioridade e tamanho. Um grupo tipico tem 5-20 issues.

O fluxo de cada issue

Issue no GitHub
↓
Especificacao (o que vai ser feito, escrito antes de codar)
↓
Implementacao (o codigo em si)
↓
Verificacao (comparar com a especificacao)
↓
Testes (funciona? nao quebra nada?)
↓
PR no GitHub (entrega linkada: “closes #N”)

Eu tenho um assistente especifico para isso — o ag-M-51. Eu digo:

Resolver issue #201

E ele faz tudo: le a issue, cria a especificacao, implementa, verifica, roda testes, cria a PR.

Trabalhando em lotes

Grupo 2 do backlog: issues #201 a #215. Tamanho medio. Resolver em sequencia, salvar depois de cada uma.

Se sao independentes, posso mandar em paralelo:

Issues #201, #202, #203 — sao independentes, pode fazer as 3 ao mesmo tempo. Cada uma em area separada para nao conflitar.

⚠️

Labels: Antes de criar issues com etiquetas, SEMPRE verifico quais existem no repositorio. Parece bobo, mas ja perdi tempo recriando issues porque etiquetas que assumi existir nao existiam.

04

Como eu construo algo do zero

Passo 1: Definir antes de construir

Quero construir um app de chamada escolar. Vai integrar com o sistema TOTVS da escola. Precisa: login do professor, lista de turmas, marcar presenca, sincronizar com o TOTVS. Me faz uma especificacao com: telas, dados, integracao, e o que pode dar errado.

A especificacao e o contrato. Sem ela, o Claude vai inventando e voce termina com algo diferente do que queria.

Passo 2: Montar a base

Agora cria o projeto seguindo a especificacao. Comeca pela estrutura: criar o projeto, instalar o que precisa, configurar o banco, criar as tabelas. Salvar depois de cada etapa.

Passo 3: Uma coisa de cada vez

Implementar o login do professor conforme a especificacao. Testar o fluxo completo.

Uma funcionalidade por vez. Testar. Salvar. Proxima. Parece lento mas e mais rapido que “faz tudo” e depois horas cacando problemas.

Passo 4: Conferir contra a especificacao

Compara o que foi feito com a especificacao original. Lista o que esta pronto, o que esta pela metade, e o que falta.

Para projetos maiores — o Orquestrador

Se o projeto e grande, uso o ag-M-00 (orquestrador). Ele e o maestro:

/ag-M-00-orquestrar novo projeto: app de gestao de tarefas. Stack: Next.js, TypeScript, Supabase, Tailwind.

Ele decide sozinho a sequencia: montar estrutura (ag-P-01) → configurar ambiente (ag-P-02) → especificar (ag-P-06) → planejar (ag-P-07) → construir (ag-B-08) → testar (ag-Q-13) → publicar (ag-D-27). Eu so dou a direcao e aprovo nos momentos-chave.

Como eu corrijo problemas

🚨

Em 193 sessoes, 71 vezes o problema foi: comecei a corrigir antes de entender o que estava errado.

O que eu faco agora

Antes de tocar em qualquer coisa, faco o Claude responder 3 perguntas:

Bug: [descricao]. ANTES de implementar qualquer coisa, me responde:
1. O que deveria acontecer?
2. O que esta acontecendo?
3. Quais sao as 3 causas mais provaveis?

So depois de revisar e concordar, digo “ok, corrige a causa #X”.

Nao resolveu. PARA. Nao tenta variacao do mesmo caminho. O diagnostico provavelmente esta errado. Reavalia.

Regra: Maximo 2 tentativas do mesmo tipo. Depois, muda a abordagem completamente.

O Orquestrador — meu assistente principal

O que e o ag-M-00

O ag-M-00 e o maestro de todo o sistema. Quando eu nao sei qual assistente usar, ou quando a tarefa e grande e envolve varios passos, eu chamo ele. Ele entende o que eu quero, decide quais assistentes usar, em que ordem, e coordena tudo.

E o assistente que eu mais uso. E tambem o mais inteligente — roda no modelo mais potente (Opus) porque precisa tomar decisoes complexas.

Como eu uso no dia a dia

Quando eu tenho uma tarefa ampla:

/ag-M-00-orquestrar implementar sistema de notificacoes com email e push. Repo em ~/Claude/GitHub/raiz-platform

Ele vai sozinho:

  1. Entender o que eu quero (classificar a intencao)
  2. Avaliar o tamanho (pequeno? medio? grande?)
  3. Decidir se precisa de especificacao primeiro
  4. Montar o plano de execucao
  5. Chamar os assistentes certos na ordem certa
  6. Verificar qualidade no final

Quando eu quero resolver issues do backlog:

/ag-M-00-orquestrar resolver issues #201 a #210 do Grupo 2

Ele analisa cada issue, decide se faz em sequencia ou paralelo, e para cada uma chama o ag-M-51 (pipeline de issues).

Quando eu nao sei por onde comecar:

/ag-M-00-orquestrar o app de chamada esta lento quando carrega a lista de turmas. O que fazer?

Ele classifica como "otimizacao", chama o ag-P-03 para investigar, depois o ag-B-11 para otimizar.

Como eu protejo meu trabalho

Salvamentos frequentes

Salvar a cada avanco que funciona. Se voce acumula 2 horas sem salvar e algo quebra, perde 2 horas. Se salva a cada 15 minutos, perde no maximo 15 minutos.

📄
Configs protegidos

Impede sobrescrever arquivos de configuracao

🔐

09

Casos reais — quando eu construo maquinas, nao tarefas

A diferenca entre usar o Claude Code para resolver UM problema e usar para construir uma MAQUINA que resolve todos os problemas daquele tipo. Dois exemplos reais.

Caso 1: O Dossier — de caos para evidencia organizada

O problema: Eu tinha milhares de mensagens de WhatsApp, dezenas de emails, e 23 contratos em PDF. Precisava transformar tudo isso num documento juridico organizado que mostrasse, cronologicamente, quais clausulas contratuais foram descumpridas — com evidencias rastreaveis ate a mensagem original.

Se eu fizesse manualmente: semanas. Se eu pedisse para o Claude "analisa essas mensagens": resultado superficial, sem rastreabilidade, sem estrutura.

O que eu fiz: Em vez de pedir a analise, eu pedi para o Claude construir um pipeline de 8 fases — uma maquina que processa qualquer volume de dados na mesma estrutura:

Fase 0: Processar os 23 contratos em 374 trechos buscaveis
↓
Fase 1: Converter WhatsApp (Android + iOS) em dados estruturados
↓
Fase 2: Converter emails em dados estruturados
↓
Fase 3: Identificar todas as pessoas (quem e quem, com apelidos)
↓
Fase 4: Classificar problemas por categoria e gravidade
↓
Fase 5: Montar linha do tempo cronologica
↓
Fase 6: Cruzar problemas com clausulas contratuais
↓
Fase 7: Montar cadeias de evidencia rastreaveis
↓
Fase 8: Gerar o documento final (Markdown + Word + PDF)

O resultado: Um dossier profissional com capa, sumario, linha do tempo, e cada afirmacao rastreavel ate a mensagem original. Se chegar nova evidencia, basta adicionar na fase 1 ou 2 e rodar o pipeline de novo.

💡

A licao: Eu nao pedi "analisa essas mensagens". Eu pedi "constroi uma maquina que transforma dados brutos em evidencia organizada". A maquina serve para este caso e para qualquer caso futuro similar. Isso e escalavel.

Caso 2: O Automata — automacao que se conserta sozinha

O problema: Eu precisava automatizar consultas em sites do governo e monitorar precos em lojas online. Qualquer automacao tradicional quebra em dias porque os sites mudam os botoes, campos, e layouts.

O que eu fiz: Em vez de pedir "automatiza o site X", eu pedi para o Claude projetar uma plataforma inteira com 3 principios:

🧠
Entender antes de agir

O sistema primeiro mapeia o site completamente — como ele funciona, quais protecoes tem, quais botoes existem

🔄
Adaptar quando falha

Se um botao muda de lugar, o sistema reencontra sozinho usando AI em vez de quebrar

🤖
Aprender com o tempo

Cada execucao alimenta uma memoria que melhora as proximas — taxas de sucesso, padroes de sites

Cada automacao e descrita como uma "receita" — nao como um script linear que quebra, mas como um fluxo com ramificacoes, alternativas, e auto-reparo. Se o caminho A falha, tenta B. Se B falha, a AI reencontra o caminho.

O resultado: Uma plataforma que qualquer pessoa pode usar para criar automacoes descrevendo em portugues o que quer ("consulta o CNPJ tal na Receita Federal"), sem precisar saber programar. E que se conserta sozinha quando os sites mudam.

💡

A licao: A diferenca entre "automatiza esse site" e "constroi uma plataforma de automacao inteligente" e a mesma diferenca entre pescar um peixe e construir uma rede. O segundo demora mais no inicio, mas resolve o problema para sempre.

O padrao que eu uso

Toda vez que tenho um problema que vai se repetir, eu me pergunto:

Eu quero resolver ESTE problema, ou quero construir uma maquina que resolve TODOS os problemas desse tipo?

Se a resposta e a segunda, eu começo com uma especificacao (ag-P-06), depois um plano (ag-P-07), e construo o sistema em fases — cada fase funcionando independente, testavel, e reutilizavel.

10

QAT — por que "funciona" nao e suficiente

O problema que testes normais nao resolvem

Testes tradicionais verificam: "O sistema funciona?" — o botao clica? a pagina carrega? o dado salva?

Mas tem uma pergunta que eles nao respondem: "O resultado e BOM?"

Exemplo: eu tenho um assistente educacional que gera planos de aula. O teste normal verifica que ele retorna um texto. Mas nao verifica se o plano e pedagogicamente adequado, se usa linguagem acessivel para o professor, se esta alinhado ao curriculo, ou se as atividades sao realmente praticaveis.

Um sistema pode funcionar perfeitamente e produzir resultados terriveis. QAT existe para pegar isso.

O que e QAT

QAT (Quality Acceptance Testing) e uma camada de testes que avalia a qualidade da saida, nao so se o sistema funciona. Em vez de um teste automatico dizer "passou" ou "falhou", uma AI avalia o resultado como um ser humano avaliaria — dando nota de 1 a 10 em criterios especificos.

Como funciona na pratica

Cada cenario de teste simula um usuario real usando o produto. Exemplo:

Cenario QAT-47: Coordenador pede atividades de alfabetizacao

Quem: Coordenador pedagogico, escola publica, 3a serie

O que pede: "Sugira 3 atividades de alfabetizacao para alunos que ainda nao leem fluentemente, alinhadas ao meu curriculo"

O que o QAT avalia (nota 1-10 em cada):

  • Aplicabilidade: O professor consegue usar amanha na sala de aula?
  • Adequacao: As atividades sao para o nivel certo (pre-fluente, nao avancado)?
  • Diferenciacao: As 3 atividades sao realmente diferentes (nao variacoes da mesma)?
  • Contextualizacao: Referencia o curriculo do professor (nao e generico)?

Nota minima para passar: 6/10 em cada criterio

As 4 camadas — economizando dinheiro

Cada cenario passa por 4 verificacoes em sequencia. Se falha numa camada mais basica, nao precisa chamar a AI (que custa dinheiro) para avaliar:

CamadaO que verificaCusto
1. BasicaA pagina carrega? Os elementos existem?R$ 0 (automatico)
2. FuncionalA resposta existe? Nao esta vazia? E do tipo certo?R$ 0 (automatico)
3. QualidadeA resposta e BOA? (AI avalia com criterios)~R$ 0,15 (1 chamada AI)
4. NegocioAtende o objetivo do usuario?~R$ 0,15 (mesma chamada)

Se a camada 1 falha (pagina nao carrega), e um problema de infraestrutura — nao precisa gastar dinheiro avaliando qualidade. Isso economiza ~30% do custo.

Por que "funciona" nao basta — um exemplo real

Exemplo: Chat educacional respondendo em ingles
O teste tradicional verificava: "retornou texto?" Sim. "Tem mais de 100 caracteres?" Sim. "Status 200?" Sim. Teste passou.

Mas o usuario perguntou em portugues e recebeu resposta em ingles. Um professor brasileiro recebendo plano de aula em ingles. Funcionou perfeitamente. Resultado inutilizavel.
O QAT pegaria isso: O criterio "Linguagem adequada" daria nota 1/10, e o cenario falharia. O teste tradicional nunca teria percebido.

O que eu aprendi errando

Cada erro aqui me custou horas reais.

“Faz tudo de uma vez”
Pedi 7 funcionalidades sem pontos de salvamento. Na 5a, algo quebrou. Perdi tudo.
Agora: Salvar depois de cada funcionalidade. Sempre.
Assistentes pisando um no outro
3 assistentes no mesmo projeto ao mesmo tempo. Corromperam o historico. 4 horas perdidas.
Agora: Cada assistente trabalha numa copia isolada. Se nao der, faco um de cada vez.
Substituicao em massa sem conferir
Trocar um texto em todos os arquivos. 28 quebraram — significados diferentes em contextos diferentes.
Agora: Lotes de 5 arquivos. Testo entre cada lote.
Corrigir sem entender
“Corrige esse erro” — 4 tentativas, nenhuma funcionou. Diagnostico errado desde o inicio.
Agora: 3 perguntas antes: O que deveria? O que esta? Quais as causas?
Credenciais vencidas
200 testes com API expirada. 45 minutos perdidos.
Agora: 1 teste rapido antes da suite toda.
Config sobrescrito
O Claude substituiu o package.json inteiro. Dependencias sumiram.
Agora: Protecao automatica que so permite edicoes pontuais.
106 protecoes = computador travou
Cada acao disparava dezenas de verificacoes. 200GB de memoria.
Agora: Poucas protecoes, todas essenciais. Nada que so “avisa”.

Como comecar

  • Semana 1 — O basico

    Instale o Claude Code. Crie um CLAUDE.md com sua stack + 3 regras. Configure permissoes. Faca uma tarefa simples. Adicione regras conforme erros acontecem. Comece a salvar frequentemente.

  • Semana 2 — Pegando o ritmo

    Instale o nivel iniciante do repositorio. Experimente o ag-M-00. Configure protecao de branch. Organize issues no GitHub. Experimente o ag-M-51 para resolver uma issue completa.

  • Semana 3 — Escalando

    Instale o nivel padrao. Crie regras separadas. Salve preferencias na memoria. Experimente assistentes em paralelo (com isolamento!).

  • Mes 1 — Produtividade alta

    CLAUDE.md maduro com regras reais. 3 protecoes automaticas. Fluxo de issue → spec → build → entrega. Licoes documentadas. Assistentes para tarefas comuns.

⭐

O conselho mais importante: Nao tente montar tudo de uma vez. Comece com o CLAUDE.md e va adicionando conforme sente necessidade. O ciclo e: usar → errar → documentar → nao repetir.

Instalacao rapida (1 comando)

curl -fsSL https://raw.githubusercontent.com/andregusman-raiz/a-gusman-claude/main/install.sh | bash

Isso faz 3 coisas automaticamente:

  1. Clona o repositorio em ~/.gusman-claude/ (a fonte da verdade)
  2. Cria symlinks em .claude/ — agents/, skills/, rules/, hooks/, shared/
  3. Instala hook de auto-update que atualiza a cada inicio de sessao (~1h de throttle)

Auto-update: como funciona

Toda vez que voce inicia o Claude Code, um hook SessionStart roda em background:

  • Executa git pull --ff-only silenciosamente
  • Maximo 1 vez por hora (nao sobrecarrega o GitHub)
  • Nunca bloqueia o inicio da sessao
  • Para desativar: bash install.sh --no-auto-update ou delete .claude/hooks/auto-update.sh
